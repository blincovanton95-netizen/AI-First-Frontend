**Проект:** CryptoTracker + AI Assistant

## 🏗️ 1. Архитектурный итог: почему fullstack, а не только фронтенд?

Создание даже простого AI-чата **почти всегда требует собственного сервера** по 5 фундаментальным причинам:

### 🔐 1. Безопасность API-ключей
Клиентский JavaScript **всегда виден пользователю**. Встраивание ключа OpenAI в `main.js` = его кража за 30 секунд через DevTools → Sources. Последствия:
- Счета на тысячи долларов за чужие запросы
- Блокировка аккаунта OpenAI
- Юридические риски при генерации запрещённого контента через ваш ключ

**Решение:** ключ живёт только в `process.env.OPENAI_API_KEY` на сервере.

### 🛡️ 2. Валидация и санитизация
Клиентскую валидацию можно обойти за 1 минуту через Postman. Только сервер может гарантировать:
- Лимит длины сообщения (защита от DoS через огромные промпты)
- Rate limiting per IP/user
- Фильтрацию prompt injection ("ignore previous instructions")
- Удаление PII (паспорта, карты) перед отправкой в LLM

### 📡 3. Streaming-оркестрация
SSE-поток от OpenAI требует:
- Установки правильных заголовков (`Content-Type: text/event-stream`, `X-Accel-Buffering: no`)
- Буферизации и транслации чанков
- Обработки отключения клиента (чтобы не платить за генерацию, которую никто не читает)
- Fallback при ошибке API

Браузер не может надёжно управлять всем этим в одиночку из-за CORS и ограничений fetch API.

### 🧠 4. Обогащение контекста
Сервер добавляет то, чего клиент не должен знать:
- System prompt с ролью ассистента
- Текущий контекст пользователя (какие монеты на экране)
- Историю диалога из БД (а не только из sessionStorage)
- Персонализацию на основе профиля

### 💰 5. Монетизация и аналитика
Только сервер может:
- Считать токены и биллить пользователей
- Логировать запросы для аудита
- Кэшировать одинаковые вопросы (экономия 50-60% расходов)
- A/B тестировать разные модели

> 💡 **Итог:** Фронтенд без бэкенда в AI-чате — как банк без замка на двери. Технически можно, но недолго и опасно.

## 📈 2. Оценка сложности промптов: UI vs Backend

### Сравнительный анализ

| Аспект | UI-компонент (ChatWidget.vue) | Backend-прокси (api/chat.js) |
|--------|-------------------------------|------------------------------|
| **Количество итераций до рабочего кода** | 1-2 | 3-5 |
| **Объём промпта** | ~150 слов | ~400 слов |
| **Частота правок после генерации** | Низкая (косметика CSS) | Высокая (edge cases, security) |
| **Риск критических багов** | Низкий (визуальные) | Высокий (безопасность, утечки) |
| **Необходимость пояснений ИИ** | Минимальная | Критическая |

### Где ИИ справлялся лучше ✅

**UI-компоненты:**
- 🎨 Вёрстка с CSS Grid/Flexbox — с первой попытки
- 🎭 Анимации `<Transition>` — идеально
- ♿ ARIA-атрибуты — после напоминания в промпте
- 📱 Адаптивность — отлично

**Backend:**
- 📦 Express boilerplate — идеально
- 🔧 Middleware (helmet, cors, rate-limit) — с первой попытки
- 📡 SSE streaming — правильно с первого раза (ключевой момент!)

### Где требовалось больше пояснений ❌

**UI:**
- 🔄 Автопрокрутка при streaming — ИИ забыл про `watch` на `content`
- 🧹 Очистка истории — не предложил метод `clearHistory()` без явного запроса

**Backend (главные сложности):**
1. **Формат SSE-чанков**  
   ИИ иногда путал `data: {...}\n\n` с `data: {...}\r\n\r\n` или забывал двойной `\n`. Пришлось явно указывать: *"Каждый чанк заканчивается ДВУМЯ символами новой строки"*.

2. **Обработка отключения клиента**  
   ИИ не добавил `req.on('close', ...)` в первой версии. Без этого при закрытии вкладки сервер продолжал генерировать и платить за токены. Пришлось объяснять концепцию "zombie streams".

3. **Sanitization истории**  
   ИИ санитизировал только текущее сообщение, но забывал про `history`. Злоумышленник мог вставить инъекцию в старое сообщение и обойти защиту. Пришлось явно указать: *"Санитизируй КАЖДОЕ сообщение в массиве history"*.

4. **Разделение ошибок**  
   ИИ возвращал JSON-ошибку даже после начала streaming. Это ломало SSE-клиент. Пришлось объяснить: *"Если заголовки уже отправлены — ошибка идёт как SSE-чанк, а не как HTTP-ответ"*.

### Вывод
> **ИИ — отличный "junior developer":** быстро пишет boilerplate, знает синтаксис, но не видит edge cases. Для UI этого достаточно (80% качества с первой итерации). Для backend нужны **senior-level пояснения** про безопасность, согласованность состояний и обработку ошибок.

## 🔍 3. Самая неочевидная техническая проблема

### 🏆 Проблема: "Потерянные" SSE-чанки на медленном интернете

**Симптом:**  
На Slow 3G текст появлялся "кусками" по 5-10 слов, потом пауза 2-3 секунды, потом снова кусок. Пользовательский опыт был ужасным — казалось, что чат "задумывается".

**Диагностика:**  
Chrome DevTools → Network → EventStream показал, что сервер отправляет чанки исправно каждые 30-80мс. Но фронтенд получал их пачками по 20-30 штук раз в 2 секунды.

**Корень проблемы:**  
```javascript
// ❌ БЫЛО (неправильно)
const reader = response.body.getReader();
while (true) {
  const { done, value } = await reader.read();
  // value может содержать НЕСКОЛЬКО SSE-чанков, склеенных вместе!
  const text = decoder.decode(value);
  // Парсинг text как ОДНОГО чанка → потеря данных
}

ReadableStream не гарантирует, что каждый read() вернёт ровно один SSE-чанк. На медленной сети браузер буферизует данные и отдаёт их пачками. Если парсить value как одно сообщение — часть чанков теряется.

** Решение: **

*** Реализовать буферизацию по строкам с обработкой неполных строк: ***

// ✅ СТАЛО (правильно)
let buffer = '';

while (true) {
  const { done, value } = await reader.read();
  if (done) break;

  buffer += decoder.decode(value, { stream: true });
  const lines = buffer.split('\n');
  
  // Последняя строка может быть неполной — оставляем в буфере
  buffer = lines.pop() || '';

  for (const line of lines) {
    if (!line.startsWith('data: ')) continue;
    const data = line.slice(6).trim();
    if (data === '[DONE]') { /* ... */ break; }
    
    try {
      const parsed = JSON.parse(data);
      assistantMessage.content += parsed.choices?.[0]?.delta?.content || '';
    } catch (e) {
      console.warn('Parse error:', e);
    }
  }
}

** Почему это неочевидно: **
- В туториалах по SSE часто показывают упрощённый код без буферизации
- На localhost проблема не воспроизводится (сеть слишком быстрая)
- ИИ сгенерировал "работающий" код, который ломался только в production-условиях

4. Роль разработчика в AI-First продукте
Окончательный ответ
Разработчик в AI-First мире — это "Архитектор доверия" (Trust Architect).
Это новая роль, которая объединяет 5 традиционных специализаций в единую дисциплину.
🎭 Пять граней роли

1. 🏗️ Архитектор систем (40% времени)
Проектирует контракты между фронтендом, бэкендом и LLM
Выбирает, где должна жить логика (клиент vs сервер vs edge)
Определяет потоки данных и состояния
Пример из курса: решение использовать Vercel Serverless вместо отдельного Node.js-сервера для monorepo-архитектуры

2. 🎯 Промпт-инженер (20% времени)
Формулирует ТЗ для ИИ как для junior-разработчика
Проектирует system prompts для LLM-агентов
Оптимизирует промпты для качества/стоимости/латентности
Пример из курса: промпт для api/chat.js с явным указанием SSE-формата, sanitization и обработки отключений

3. 🔌 Интегратор (20% времени)
Связывает разнородные системы (Vue + OpenAI + Vercel + Redis)
Обеспечивает совместимость форматов данных
Настраивает CI/CD для AI-компонентов
Пример из курса: интеграция streaming-ответа OpenAI с Vue-реактивностью через ReadableStream + watch

4. 🛡️ Специалист по безопасности (15% времени)
Защищает API-ключи от утечки
Предотвращает prompt injection атаки
Обеспечивает GDPR compliance (фильтрация PII)
Настраивает rate limiting и валидацию
Пример из курса: sanitization сообщений + rate limit 20 req/min + валидация длины

5. ⚡ Специалист по производительности (5% времени)
Оптимизирует CWV (LCP, INP, CLS) для AI-интерфейсов
Настраивает кэширование и lazy loading
Мониторит стоимость LLM-вызовов

Пример из курса: preload API, скелетоны для CLS, кэширование ответов в Redis

📊 Почему именно "Архитектор доверия"?
Потому что главная задача AI-First разработчика — сделать вероятностную систему предсказуемой и безопасной для пользователя.

| Источник недоверия | Как разработчик его устраняет |
|--------------------|-------------------------------|
|"ИИ может украсть мои данные" | Шифрование, PII-фильтрация, прозрачная политика |
| "ИИ может ответить ерунду" | Guardrails, валидация, fact-checking |
| "ИИ может сломать интерфейс" | A11y, graceful degradation, fallback на GUI |
| "ИИ может списать деньги" | Rate limiting, cost monitoring, budget alerts
| "ИИ работает медленно" | Streaming, preload, кэширование, edge computing |
