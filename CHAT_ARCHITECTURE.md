# Архитектура AI-чат виджета: CryptoTracker Assistant

## Цель
Встроить в приложение CryptoTracker интеллектуального помощника, который:
- Отвечает на вопросы о криптовалютах на естественном языке
- Анализирует текущий контекст пользователя (какие монеты на экране)
- Потоково отдаёт ответы (streaming) для "живого" UX
- Сохраняет историю диалога в рамках сессии

---

## Компоненты архитектуры

### Фронтенд (Vue 3)

#### `ChatWidget.vue`
- Плавающая кнопка в правом нижнем углу (`position: fixed`)
- Переключает видимость `ChatWindow` по клику
- Показывает бейдж с количеством непрочитанных сообщений
- Поддерживает горячую клавишу `Ctrl+K`

#### `ChatWindow.vue`
- Заголовок с названием ассистента и кнопкой закрытия
- Область сообщений (автопрокрутка вниз при новых)
- Компонент ввода внизу
- Обработка `Escape` для закрытия, фокус-трап для a11y

#### `MessageBubble.vue`
- Пропсы: `role` ('user' | 'assistant' | 'system'), `content`, `timestamp`, `isStreaming`
- Разная стилизация для user/assistant (цвета, выравнивание)
- Поддержка Markdown-рендеринга через `marked` (для ответов ассистента)
- Анимация появления через `<Transition>`

#### `useChat.js` (Composable)
- Состояние: `messages`, `isLoading`, `error`, `isStreaming`
- Методы: `sendMessage(text)`, `clearHistory()`, `cancelStream()`
- Подписка на SSE через `EventSource` или `fetch` с `ReadableStream`
- Локальное хранение истории в `sessionStorage`

---

### Бэкенд (Node.js + Express)

#### Критическая роль прокси-сервера:

1. ** Хранение секретов**  
   API-ключ OpenAI хранится **только** в `process.env.OPENAI_API_KEY` на сервере. Клиент никогда его не видит.

2. ** Валидация и санитизация**  
   - Проверка длины сообщения (защита от DoS через огромные промпты)
   - Rate limiting (защита от флуда и списания средств)
   - Sanitization (удаление попыток prompt injection: "ignore previous instructions...")
   - Фильтрация запрещённого контента

3. ** Streaming-оркестрация**  
   - Принимает SSE-поток от OpenAI
   - Транслирует его клиенту через `text/event-stream`
   - Может модифицировать поток (например, добавлять метаданные)
   - Обрабатывает ошибки API и возвращает дружелюбные сообщения

4. ** Обогащение контекста**  
   - Добавляет system prompt с ролью ассистента
   - Инжектирует текущий контекст пользователя (например, "пользователь смотрит на Bitcoin и Ethereum")
   - Управляет историей диалога (последние N сообщений)

┌─────────────┐ HTTP POST ┌─────────────────┐ HTTPS ┌──────────────┐
│ │ {message, │ │ {api_key, │ │
│ Frontend │ context} ───▶ │ Proxy Server │ ──────────▶ │ OpenAI API │
│ (Vue 3) │ │ (Node/Express) │ │ │
│ │ │ │◀────────── │ │
│ │ │ │ SSE stream │ │
└──────┬──────┘ └────────┬────────┘ └──────────────┘
│      │                 |
│ SSE / text/event-stream │ Валидация, rate limit,
│ (побайтовая передача) │ санитизация, обогащение
       ▼                 ▼
┌─────────────────────────────────────────────────┐
│ Рендеринг сообщений в реальном времени │
│ (каждые 20-50мс появляется новый токен) │
└─────────────────────────────────────────────────┘


### Детализация шагов:

1. **Пользователь** вводит "Что думаешь о Bitcoin?" и нажимает Enter
2. **Frontend** (`useChat.js`):
   - Добавляет сообщение в локальный стейт с `role: 'user'`
   - Делает `fetch('/api/chat', { method: 'POST', body: {...} })`
   - Открывает `ReadableStream` для чтения SSE
3. **Proxy Server** (`routes/chat.js`):
   - Валидирует вход (длина, формат)
   - Проверяет rate limit по IP
   - Строит контекст: system prompt + история + текущее сообщение
   - Вызывает `openai.chat.completions.create({ stream: true, ... })`
4. **OpenAI API** возвращает SSE-поток токенов
5. **Proxy Server** транслирует каждый токен клиенту через `res.write(`data: ${JSON.stringify(chunk)}\n\n`)`
6. **Frontend** парсит SSE, обновляет `content` последнего сообщения побайтово
7. **UI** перерисовывает `MessageBubble` с `isStreaming: true` → анимация "печатает"
8. По завершении потока: `isStreaming: false`, сообщение сохраняется в историю

## Безопасность

| Угроза | Защита |
|--------|--------|
| Кража API-ключа | Ключ только на сервере, никогда не в клиенте |
| DoS через огромные промпты | `validateInput` ограничивает длину до 1000 символов |
| Флуд запросами | `rateLimit`: 10 запросов/мин на IP, 100/день на сессию |
| Prompt injection | `sanitize` удаляет паттерны типа "ignore previous instructions" |
| Утечка истории | История хранится только в sessionStorage клиента, на сервере — только в рамках запроса |
| XSS через ответы LLM | Markdown рендерится через `DOMPurify` на клиенте |

## Деплой

### Вариант 1: Monorepo на Vercel

**Плюсы:** один деплой, бесплатно, автоматический HTTPS  
**Минусы:** ограничение на время выполнения (10s на Hobby plan) — может не хватить для длинных streaming-ответов

### Вариант 2: Раздельный деплой
- Frontend → Vercel (статика)
- Backend → Railway / Render / Fly.io (Node.js сервер)
**Плюсы:** нет ограничений по времени, полный контроль  
**Минусы:** два деплоя, нужно настроить CORS

### Рекомендация для учебного проекта: **Вариант 1** (проще), с fallback на mock-режим при превышении лимитов.

## Тестирование

### Unit-тесты (Vitest)
- `useChat.js`: отправка, обработка ошибок, отмена stream
- `contextBuilder.js`: правильная сборка промпта
- `sanitize.js`: удаление инъекций

### E2E-тесты (Playwright)
- Открытие виджета по кнопке и по Ctrl+K
- Отправка сообщения → получение ответа
- Streaming: проверка, что текст появляется постепенно
- Offline: graceful degradation (показать ошибку, не ломать UI)

### Нагрузочное тестирование
- 50 параллельных запросов → проверка rate limit
- Огромный промпт (10KB) → проверка валидации
- Невалидный JSON → проверка обработки ошибок
